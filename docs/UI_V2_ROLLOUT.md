# UI v2 平滑切换方案

## 目标

新版只替换登录后界面的视觉外壳和交互表现。现有路由、API、权限、表单、筛选、导出、弹窗、国际化、新手引导及业务组件继续复用。

经典界面必须在灰度期内一直可用，且生产首次包含 UI v2 时不得自动切换任何用户。

## 开关行为

构建参数：

| 参数 | 可选值 | 默认值 | 行为 |
| --- | --- | --- | --- |
| `VITE_UI_V2_ROLLOUT_MODE` | `off` / `preview` / `percentage` / `full` | `preview` | 控制新版启用范围 |
| `VITE_UI_V2_ROLLOUT_PERCENT` | `0` - `100` | `0` | `percentage` 模式的稳定用户比例 |

模式说明：

- `off`：强制所有用户使用经典界面，忽略新版偏好。
- `preview`：默认经典界面；只有访问 `?ui=v2` 的当前账号启用新版。
- `percentage`：没有手动偏好的账号按用户 ID 稳定分桶；同一账号不会反复跳组。
- `full`：默认新版；主动回退过的账号仍保留经典界面。

账号级入口：

- 启用新版：登录后访问 `/dashboard?ui=v2`。
- 回退经典版：点击新版顶栏的返回按钮，或访问任意登录后页面并添加 `?ui=legacy`。
- 选择会按账号保存在当前浏览器中，不会改变其他用户。

## 推荐发布阶段

### 阶段 0：本地和测试环境

1. 对比经典版与新版的核心流程。
2. 覆盖 `/dashboard`、`/keys`、`/usage`、`/profile` 以及管理员高频页面。
3. 检查桌面、手机、亮色、暗色、键盘操作和降低动态效果模式。

### 阶段 1：生产内部预览

使用默认构建参数：

```bash
scripts/deploy-prod.sh \
  --host sub2api-prod \
  --ui-v2-rollout-mode preview \
  --ui-v2-rollout-percent 0
```

发布后所有现有用户仍使用经典界面。内部测试账号通过 `/dashboard?ui=v2` 调试至少一个完整业务周期。

### 阶段 2：小比例灰度

内部预览稳定后，使用 `percentage` 模式从 5% 开始：

1. 5%，至少观察 24 小时。
2. 25%，至少观察 24 小时。
3. 50%，至少观察 48 小时。
4. 100% 灰度，但继续保留经典界面回退入口。

每次只调整比例，不同时混入无关功能改动。以 5% 灰度为例：

```bash
scripts/deploy-prod.sh \
  --host sub2api-prod \
  --ui-v2-rollout-mode percentage \
  --ui-v2-rollout-percent 5
```

### 阶段 3：默认新版

确认 100% 灰度稳定后改为 `full`。经典界面至少再保留一个发布周期，之后再单独评估是否删除旧样式。

## 每阶段验收项

- 登录、退出、Token 刷新和权限跳转无变化。
- 用户与管理员菜单项目数量、可见条件和目标路由无变化。
- API 密钥创建、编辑、复制、暂停和删除无变化。
- 用量筛选、分页、列设置、导出和错误详情无变化。
- 充值、订阅、订单、兑换和个人资料流程无变化。
- 前端错误率、API 4xx/5xx 比例和页面加载 P95 不劣于经典版。
- 无横向溢出、遮挡、不可关闭面板或移动端安全区问题。

## 停止灰度与回滚

出现下列任一情况立即停止扩量：

- 核心操作不可达或权限展示错误。
- 新版前端错误率明显高于经典版。
- 关键页面 P95 加载时间持续恶化。
- 移动端出现大面积布局、滚动或手势故障。

回滚顺序：

1. 单账号问题先使用顶栏返回按钮切回经典界面。
2. 灰度问题停止扩量，并重新构建 `off` 模式镜像。
3. 严重问题直接部署发布前保留的生产镜像。

UI v2 不包含数据库迁移，回滚前端镜像不会回滚或丢失业务数据。
